# 二、仍然存在的“不可落地 / 难落地”点（必须严肃对待）

---

## ❗ 1. slow 统计仍有一个隐藏歧义

位置：12.1 sync_fast 

你已明确：

* 按 command signature
* 最近 2 次
* 不跨 day

但仍缺一条关键规则：

> “最近 2 次”是成功 + 失败都统计，还是只统计成功？

当前没有定义 slow 是否只对 exit_code=0 的命令统计。

这会导致两种实现分叉：

实现 A：

```
只统计成功命令
```

实现 B：

```
统计所有执行（包括失败）
```

结果不同。

---

### 建议（必须补充一句）：

```text
slow 统计仅针对 exit_code=0 的命令执行。
失败命令不进入 slow 统计窗口。
```

否则实现将出现差异。

---

## ❗ 2. “非平凡 diff >=10 行”在 monorepo 下仍有风险

位置：8.6 新证据判定 

当前规则：

* 仅统计白名单路径
* numstat 合计 >=10

问题：

如果：

* 修改 9 行真实业务逻辑
* 触发重大语义变化
* 但 <10 行

会被视为“无新证据”

这会误触发熔断。

---

### 建议（二选一）

### ✅ 推荐（更稳定）

改为：

```
只要白名单路径下存在 tracked 文件变化，即视为新证据。
不再使用 >=10 行阈值。
```

### ❌ 或者

降低为：

```
>=5 行
```

> =10 行对日常改动来说过高。

---

## ❗ 3. window_seconds 触发条件未与 signature 强绑定

位置：8.2 熔断触发条件 

当前写法：

> 在 window_seconds 内，参数完全重复且失败次数 >=3

问题：

“参数完全重复”如何判断？

你已有 signature 机制。

这里若再引入“参数完全重复”语义判定，会出现：

* signature 相同但参数不同？
* 或反之？

---

### 建议（强收敛）

删除：

> 参数完全重复

改为：

```
在 window_seconds 内，同 signature 且同 digest 失败次数 >=3
```

统一口径到 signature + digest，不再额外引入“参数完全重复”。

### 分叉点 2：diff 行数统计是否累计 rename/delete

你写：

```
git diff --numstat HEAD
```

但 numstat 格式为：

```
added deleted filename
```

* 是否统计 deleted？
* rename 情况 numstat 显示 0 0

未写明。

---

### 建议补一句：

```

rename 不单独计入
```
