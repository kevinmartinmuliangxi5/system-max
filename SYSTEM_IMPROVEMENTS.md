## 💡 改进方案2: Brain智能任务拆解

### 当前状态

Brain已经在做任务拆解，输出的 `project_state.json` 就是拆解后的任务列表：

```json
{
  "project_name": "...",
  "blueprint": [
    {
      "task_name": "任务1",
      "instruction": "...",
      "status": "PENDING"
    },
    {
      "task_name": "任务2",
      "instruction": "...",
      "status": "PENDING"
    }
  ]
}
```

### 问题

1. **拆解粒度不够智能**
   - 简单任务也拆成多步（浪费）
   - 复杂任务拆得不够细（质量差）

2. **缺少依赖关系**
   - 任务之间的依赖不明确
   - 可能导致执行顺序错误

3. **缺少验证步骤**
   - 修改后没有验证
   - 容易遗漏bug

---

### 解决方案: 智能拆解算法

#### 拆解原则

**简单任务（1-2步）**:
- 单文件的小改动
- 删除功能
- 简单配置修改

**示例**:
```
需求: "删除侧边栏"
拆解:
  1. 修改所有页面文件添加config
  （不拆解，一步完成）
```

**中等任务（3-5步）**:
- 新增简单功能
- 多文件修改
- 需要测试验证

**示例**:
```
需求: "添加用户登录功能"
拆解:
  1. 创建登录页面UI
  2. 实现登录逻辑
  3. 添加session管理
  4. 测试登录流程
```

**复杂任务（6-15步）**:
- 新建重要模块
- 架构级修改
- 需要多轮验证

**示例**:
```
需求: "实现完整的用户系统"
拆解:
  1. 设计数据库表结构
  2. 创建用户模型
  3. 实现注册功能
  4. 实现登录功能
  5. 实现密码加密
  6. 实现session管理
  7. 实现权限验证
  8. 创建用户管理页面
  9. 添加单元测试
  10. 集成测试
  11. 性能测试
  12. 文档编写
```

---

#### 拆解算法

**在 Brain 提示词中添加**:

```markdown
## 📊 智能任务拆解协议

### 拆解粒度判断

根据任务复杂度确定拆解粒度：

#### 1. 简单任务（不拆解或1-2步）

**判断标准**:
- 单文件小改动（<50行代码）
- 配置修改
- 删除功能
- 简单的样式调整

**处理**:
```json
{
  "blueprint": [
    {
      "task_name": "删除侧边栏",
      "instruction": "在所有页面文件开头添加st.set_page_config(initial_sidebar_state='collapsed')",
      "target_files": ["app.py", "pages/2_🎁_赠君.py"],
      "priority": "HIGH",
      "status": "PENDING"
    }
  ]
}
```

#### 2. 中等任务（3-5步）

**判断标准**:
- 新增简单功能
- 多文件协同修改
- 需要基本测试
- 改动量50-200行

**处理**:
```json
{
  "blueprint": [
    {
      "task_name": "步骤1: 创建UI组件",
      "instruction": "...",
      "target_files": ["..."],
      "priority": "HIGH",
      "status": "PENDING",
      "dependencies": []
    },
    {
      "task_name": "步骤2: 实现业务逻辑",
      "instruction": "...",
      "target_files": ["..."],
      "priority": "HIGH",
      "status": "PENDING",
      "dependencies": ["步骤1"]
    },
    {
      "task_name": "步骤3: 集成测试",
      "instruction": "...",
      "target_files": ["..."],
      "priority": "MEDIUM",
      "status": "PENDING",
      "dependencies": ["步骤2"]
    }
  ]
}
```

#### 3. 复杂任务（6-15步）

**判断标准**:
- 新建重要模块/功能
- 架构级改动
- 需要多层验证
- 改动量>200行

**拆解策略**:

**Phase 1: 基础搭建（2-3步）**
- 数据模型/接口设计
- 基础架构搭建
- 核心功能框架

**Phase 2: 功能实现（3-6步）**
- 各子功能逐一实现
- 每个子功能都是独立可测试的

**Phase 3: 测试验证（2-3步）**
- 单元测试
- 集成测试
- 性能/安全测试

**Phase 4: 优化完善（1-2步）**
- 代码优化
- 文档完善

**示例**:
```json
{

      "instruction": "...",
      "phase": 2,
      "priority": "HIGH",
      "status": "PENDING",
      "dependencies": ["Phase 1.2"]
    },
    {
      "task_name": "Phase 2.2: 实现登录功能",
      "instruction": "...",
      "phase": 2,
      "priority": "HIGH",
      "status": "PENDING",
      "dependencies": ["Phase 1.2"]
    },
    // ... 更多功能

    // Phase 3

### 拆解的好处

#### 1. 质量提升

**细粒度拆解**:
```
任务: "实现用户系统"

❌ 不拆解:
  → Worker一次实现所有功能
  → 容易遗漏边界情况
  → 测试不充分
  → 质量差

✅ 拆解成12步:
  → 每步专注一个小功能
  → 每步都有验证
  → 逐步积累，质量高
  → 出问题容易定位
```

#### 2. 进度可控

```
不拆解:
  Loop 1: 实现用户系统... [50%?]
  Loop 2: 继续实现... [80%?]
  Loop 3: 还在实现... [95%?]
  （不知道还要多久）

拆解:
  Loop 1: Phase 1.1 完成 ✅ [8%]
  Loop 2: Phase 1.2 完成 ✅ [16%]
  Loop 3: Phase 2.1 完成 ✅ [25%]
  （进度清晰，可预测）
```

#### 3. 易于中断和恢复

```
不拆解:
  中断后不知道做到哪了
  恢复时要重新理解上下文

拆解:
  中断后看蓝图就知道进度
  恢复时直接继续下一步
```

#### 4. 并行开发（未来）

```
如果有多个Worker:
  Worker A: Phase 2.1 注册功能
  Worker B: Phase 2.2 登录功能
  （并行开发，提速）
```

### 工作原理

**代码实现** (ralph_auto_stream_fixed.sh):

```bash
# 1. 信号捕获
trap handle_interrupt SIGINT

# 2. 中断处理函数
handle_interrupt() {
    echo "请选择操作："
    echo "  f / feedback   → 提供反馈意见"

    read -p "👉 您的指令: " user_command

    case "$user_command" in
        f|feedback)
            echo "💬 输入反馈（输入'END'结束）："
            > "$user_feedback_file"  # .ralph/user_feedback.txt
            while true; do
                read -p "  " feedback_line
                if [ "$feedback_line" = "END" ]; then
                    break
                fi
                echo "$feedback_line" >> "$user_feedback_file"
            done
            echo "✅ 反馈已保存"
            ;;
    esac
}

# 3. 在下一轮整合反馈
if [ -f "$user_feedback_file" ]; then
    echo "💬 检测到用户反馈，整合到指令中..."
    echo "" >> .ralph/current_instruction.txt
    echo "## 用户反馈（必须优先处理）：" >> .ralph/current_instruction.txt
    cat "$user_feedback_file" >> .ralph/current_instruction.txt
    rm -f "$user_feedback_file"
fi
```
