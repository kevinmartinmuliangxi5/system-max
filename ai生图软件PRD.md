# AI生图应用 PRD（MVP v3.1，可直接审查与交付 Codex）

- 文档状态：可开工版本
- 更新日期：2026-02-26
- 适用范围：MVP（iOS + Android）
- 核心约束：MVP 运行方案除硅基流动 API 调用外，不引入任何付费云基础设施

---

## 0. 需求收敛（先计划）

### 0.1 目标（Goals）

1. 做一款手机端 AI 生图/改图应用，核心体验是“多轮对话 + 局部涂抹重绘”。
2. 方案可直接给审查和 Codex 执行，避免二次解释。
3. 成本约束：MVP 技术运行成本仅包含硅基流动 API 调用成本。
4. UI 开发过程由 Codex 按 Pencil MCP 读取 `.pen` 文件 1:1 落地。

### 0.2 非目标（Non-Goals）

1. 不做云数据库、云对象存储、云鉴权系统。
2. 不做 Web 端。
3. 不做复杂运营后台。
4. 不做训练/微调模型。

### 0.3 约束（Constraints）

1. 仅使用 SiliconFlow API 作为模型能力来源。
2. 客户端本地持久化：`expo-sqlite` + `expo-file-system` + `MMKV`。
3. 发布态禁止内置共享生产 Key。
4. 所有 AI 响应先过 Schema 校验再入库。

### 0.4 风险（Top Risks）

1. Key 泄露风险（发布包逆向）。
2. Base64 入全局状态导致 OOM。
3. API 高峰期 429/503 导致体验波动。
4. 移动端图像编辑性能不足。

### 0.5 验收标准（Acceptance）

1. 新用户 3 步内完成首张图生成。
2. 局部重绘链路可用：上传/生成 -> 涂抹 -> 指令 -> 返回结果。
3. 出错可恢复：429/503/超时均有重试与统一提示。
4. 本地数据可回放最近 N 次请求摘要（脱敏）。
5. 有成本日志：每次生成记录估算消耗、耗时、状态码。

## 1. 审查问题清单逐条处理（采纳决策）

### 1.1 P0-1 客户端共享 Key 风险

- 审查意见：采纳。
- 决策：
1. 开发态：允许 `.env` 读取测试 Key。
2. 发布态：默认 BYOK（用户输入自己的 SiliconFlow Key，本地加密存储）。
3. 红线：发布包禁止内置共享生产 Key。
- 落地章节：第 4 章。

### 1.2 P0-2 “100% 零成本”绝对表述

- 审查意见：部分采纳（表述修正，策略不变）。
- 决策：
1. 技术运行侧：坚持“仅硅基流动 API 成本，无其他云基础设施成本”。
2. 经营侧：分发资质/签名/人力不计入技术方案成本，但在风险边界中单列。
- 落地章节：第 12 章。

### 1.3 P1-1 版本策略静态化

- 审查意见：采纳。
- 决策：增加 `Minimum / Recommended / Rollback` 三段版本治理，并给出兼容矩阵。
- 落地章节：第 5 章。

### 1.4 审查建议新增项（全部采纳）

1. 每个 Phase 补 DoD：输入/输出/验收命令/回滚。
2. 自动护栏：禁止 Base64 写入 store。
3. `contracts/`：AI 响应 Schema 校验。
4. 失败回放：本地保存最近 N 次脱敏摘要。
5. 成本日志：记录估算消耗和失败码。

## 2. 产品定义

### 2.1 产品定位

- 名称：画境（MVP）
- 价值主张：让用户在手机上用自然语言和涂抹交互快速生成/修改图片。

### 2.2 目标用户

1. 社媒创作者（高频配图）。
2. 轻设计需求用户（封面、海报、头像）。

### 2.3 核心用户旅程

1. 进入应用 -> 输入提示词 -> 生成第一张图。
2. 点开图片 -> 涂抹局部 -> 输入“改成…” -> 返回局部重绘结果。
3. 保存图片（带 AIGC 标识）到相册。

---

## 3. MVP 功能范围

### 3.1 In Scope

1. 文生图。
2. 图生图（以现有图为输入）。
3. 局部涂抹重绘（mask）。
4. 多轮对话历史（本地）。
5. 本地图库与导出。
6. 错误重试与统一提示。
7. 成本日志与失败回放。

### 3.2 Out of Scope

1. 云同步与多设备同步。
2. 社区广场。
3. 团队协作。
4. 在线支付系统（MVP 不做共享商业化闭环）。

## 4. 安全与密钥策略（发布红线）

### 4.1 双态策略

1. 开发态（Dev）
- 使用 `.env` 测试 Key。
- 仅用于本地调试包，不可上架。

2. 发布态（Release）
- BYOK：用户手动输入 SiliconFlow Key。
- Key 使用系统安全存储（iOS Keychain / Android Keystore）。
- 仅本机使用，不上传业务服务端。

### 4.2 强制红线

1. 发布包中禁止硬编码共享生产 Key。
2. 日志禁止输出完整 Key，仅允许显示前 3 后 2 位掩码。
3. 网络层请求失败日志需脱敏。

## 5. 技术栈与版本治理

### 5.1 技术栈（MVP）

1. Expo + React Native + TypeScript。
2. 状态：Zustand（轻状态）+ SQLite（历史数据）+ MMKV（配置）。
3. 网络：Axios + Retry Interceptor。
4. 图形：Skia + Reanimated + Gesture Handler。
5. 校验：Zod（或 JSON Schema 校验库）。

1. Minimum

3. Rollback
- 保留“上一稳定矩阵”分支。
- 若升级后出现阻断问题，24 小时内回退到上一矩阵。

## 6. 系统架构（零云基础设施）

### 6.1 架构原则

1. 无业务后端。
2. 业务数据本地存储。
3. API 直连 SiliconFlow。

### 6.2 数据流

1. 用户输入 -> Prompt 处理 -> SiliconFlow 请求。
2. 响应图像（Base64）立即落盘文件。
3. 数据库只存 `file://` 路径与元数据。
4. UI 渲染本地 URI。

### 6.3 OOM 防线（硬规则）

1. 全局 store 禁止保存 Base64 图像。
2. Base64 仅在“请求返回到落盘完成”短生命周期存在。
3. 大图进入编辑前先降采样。

## 7. API 与契约（contracts）

### 7.1 API 清单

1. Prompt 增强：`POST /v1/chat/completions`（DeepSeek-V3 via SiliconFlow）。
2. 图像生成/编辑：`POST /v1/images/generations`（FLUX.1-Kontext-pro via SiliconFlow）。

### 7.2 contracts 目录要求

- 目录：`contracts/`
- 文件（示例）：
1. `prompt-enhance.response.schema.json`
2. `image-generation.response.schema.json`
3. `error.schema.json`

### 7.3 契约执行规则

1. 响应先校验 Schema，再写库。
2. 校验失败视为可重试错误，进入统一错误流程。
3. 任何未通过校验的数据不得进入 UI 状态。

### 7.4 错误与重试映射

| 场景 | 策略 | UI 文案 |
|---|---|---|
| 429 | 指数退避重试（最多 3 次） | 请求较多，已自动重试 |
| 503 | 指数退避重试（最多 3 次） | 服务繁忙，正在重试 |
| 超时 | 重试 2 次 | 网络波动，请稍后再试 |
| 4xx 非 429 | 不重试 | 请求参数异常，请修改后再试 |
| 契约校验失败 | 1 次重试 | 返回数据异常，正在重试 |

## 8. 数据模型与存储设计

### 8.1 本地数据库表（最小集）

1. `conversations`
- `id`, `title`, `created_at`, `updated_at`

2. `messages`
- `id`, `conversation_id`, `role`, `text`, `image_uri`, `mask_uri`, `status`, `created_at`

3. `request_logs`
- `id`, `request_type`, `status_code`, `retry_count`, `duration_ms`, `cost_estimate`, `created_at`

4. `replay_logs`
- `id`, `request_type`, `masked_prompt`, `error_code`, `created_at`

### 8.2 回放与日志策略

1. 回放日志仅存最近 N=50 条。
2. Prompt 与 Key 全部脱敏。
3. 失败日志可在“调试页”查看并一键复制（脱敏后）。

## 9. UI/UX 规范（Pencil MCP 强约束）

### 9.1 开发流程硬规则

1. 先用 Pencil MCP 读取 `.pen` 文件。
2. 按设计稿还原布局、间距、字号、状态态。
3. 不允许“凭感觉”生成 UI。

### 9.2 核心页面

1. 对话主页面（输入区 + 消息流 + 图片卡片）。
2. 全屏编辑器（缩放/平移/涂抹/撤销/重做）。
3. 结果页（保存、重试、继续编辑）。

### 9.3 状态态

1. 加载中（非系统默认菊花，使用自定义脉冲动画）。
2. 错误态（有原因、有下一步动作）。
3. 空态（引导用户完成首图）。

---

## 10. 合规与内容标识（MVP 必做）

1. 显式标识：导出时叠加“AI 生成 / AIGC”可见标记。
2. 隐式标识：写入 EXIF（包含 `AIGC` 字样和模型信息）。
3. 内容安全：
- 请求前关键词拦截（本地规则）。
- 结果异常可举报（本地记录）。

---

## 11. 性能与可靠性指标

1. 首屏可交互时间：< 2.5s（中端机）。
2. 普通生成 P50：< 8s。
3. 涂抹操作帧率：目标 55-60 FPS。
4. 崩溃率：< 0.5%。

## 14. 开发阶段计划（含 DoD 与回滚）

### Phase 0：工程骨架与约束落地

- 输入：PRD、设计稿、模型 API 信息
- 输出：项目骨架 + contracts + retry + logging 基础设施
- 验收：`test/typecheck/lint/build` 全通过
- 回滚：回退到骨架初始化 tag

### Phase 1：文生图 + 对话流

- 输入：Phase 0 输出
- 输出：首图生成可用，历史可回看
- 验收：首图路径 E2E 通过
- 回滚：关闭图片入口，仅保留文本对话

### Phase 2：局部重绘编辑器

- 输入：Phase 1 输出
- 输出：mask 生成、重绘、结果替换
- 验收：编辑链路 E2E 通过；FPS 达标
- 回滚：降级为“框选重绘”无自由涂抹

### Phase 3：合规导出 + 稳定性

- 输入：Phase 2 输出
- 输出：显式/隐式标识导出、完整错误态、回放页
- 验收：导出与异常路径回归全绿
- 回滚：暂时关闭导出，只允许应用内查看

## 16. 风险清单与应对

1. 用户不愿 BYOK：
- 应对：引导文案 + Key 检查 + 导入历史。

2. 直连 API 波动：
- 应对：重试、超时兜底、可重入操作。

3. 设备性能差异：
- 应对：图片降采样、编辑层简化、低端机降级。

4. 合规认知偏差：
- 应对：保留标识与日志，审查前固定证据包。
