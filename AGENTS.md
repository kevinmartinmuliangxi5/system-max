# Codex 单脑 SOP（简化版）

本文件用于在 Codex 中统一执行流程，不再区分主脑/副脑。

## 0. 总原则（必须遵守）

1. 先计划再实现：先明确目标、边界、验收标准，再动手改代码。
2. 先证据后结论：没有测试/日志/构建结果，不宣称“已完成”。
3. 先根因后修复：Bug 必须先复现并定位根因，再提交修复。
4. 小步快跑：每次只做一个可验证改动，持续反馈进度。

## 1. 标准执行流（单脑 7 步）

1. 需求收敛
- 输出：目标、非目标、约束、风险、验收标准。

2. 计划拆解
- 把任务拆成 3-7 个步骤，每步都定义验证方式。
- 验证方式：单测、集成测试、构建、截图、接口回放、日志比对。

3. 先写失败测试（TDD-RED）
- 为目标行为写测试并确认失败。

4. 写最小实现（TDD-GREEN）
- 只写让当前失败测试通过的最小改动。

5. 重构与清理（TDD-REFACTOR）
- 消除重复、提升可读性，保持测试全绿。

6. 完成前验证
- 至少执行：测试 + Lint/类型检查 + 构建（若项目有）。
- 关键路径手工验证一次（例如登录、核心 API、主页面）。

7. 交付与沉淀
- 输出变更摘要、验证证据、剩余风险、后续建议。
- 把可复用经验写入 `docs/session-notes.md`。

## 2. 触发词（你可以直接这样下指令）

### 2.1 计划类
- “写实现计划”
- “拆成可执行任务”
- “先给验收标准再做”

### 2.2 TDD 类
- “用 TDD 做”
- “先写失败测试”
- “按 red-green-refactor 执行”

### 2.3 调试类
- “系统化调试这个 bug”
- “先找根因再修”
- “给我最小复现和假设验证”

### 2.4 完成验证类
- “完成前做全量验证”
- “给证据，不要口头完成”
- “输出风险清单”

## 3. 完成验收清单（DoD）

每次任务完成前，至少满足以下条目：

- [ ] 目标功能与需求一致，无越界修改。
- [ ] 测试已通过（新增/受影响测试）。
- [ ] 静态检查通过（lint/typecheck，如项目有）。
- [ ] 构建通过（build，如项目有）。
- [ ] 关键场景已手工验证并有证据（截图/日志/命令结果）。
- [ ] 无明显安全回归（鉴权、输入校验、敏感信息处理）。
- [ ] 输出了“变更说明 + 验证证据 + 剩余风险”。

## 5. MCP 使用策略（简版）

1. Context7：查“最新官方文档/API 变化”时优先使用。
2. Chrome DevTools MCP：前端页面截图、控制台、网络、性能分析优先使用。
3. Pencil MCP：涉及 `.pen` 设计文件读取/修改/截图时使用。

## 6. 推荐提问模板（高效）

### 新功能
“请按本 SOP 执行：先计划，再 TDD 实现，再完成前验证。目标是：<目标>。验收标准：<标准>。”

### 修 Bug
“请按系统化调试模板处理这个问题：<现象>。先根因再修复，并补回归测试。”

### 代码审查
“请做 review，优先找 bug/风险/回归，给出文件定位和修复建议，再给简短总结。”

## 7. 禁止事项

1. 未定位根因就直接修 bug。
2. 没有失败测试就直接写生产代码（可测试场景下）。
3. 无证据直接宣称“已完成”。
4. 未经确认执行破坏性操作（如强制 reset/删除关键数据）。

## 8. Brainstorm 前置检查（强制）

1. 每做一步工作前，必须先执行“是否需要 brainstorm”判断。
2. 判断规则：
- 只要涉及需求理解、方案设计、功能新增、行为修改、文档结构重组，默认需要先 brainstorm。
- 仅当任务是纯机械执行（如已明确命令安装、纯拷贝、无设计决策的格式化）时，可跳过 brainstorm。
3. 执行要求：
- 若需要 brainstorm：先完成需求收敛（目标/非目标/约束/风险/验收标准），再进入实现。
- 若判断可跳过：必须在执行前写明“跳过理由”，并确保不引入新设计决策。
4. 禁止事项：
- 未完成 brainstorm 判断，禁止直接开始实现或改写核心文档。
- 把“任务简单”作为跳过 brainstorm 的唯一理由。

## 9. Brainstorm 选项输出规范（强制）

在 brainstorm 阶段，凡是给出多方案/多选项，必须同时提供以下三项信息：
1. 选项优劣比较：每个选项都要写清主要优点与主要缺点。
2. 推荐选择：明确给出首选方案，不可只罗列不决策。
3. 推荐原因：说明推荐依据（目标匹配度、风险、实施成本、验收可行性）。

执行格式要求：
1. 选项数量建议 2-3 个，避免过多导致决策负担。
2. 先给推荐方案，再给完整对比。
3. 若无法推荐，必须说明缺失信息并提出下一条单问题澄清。
