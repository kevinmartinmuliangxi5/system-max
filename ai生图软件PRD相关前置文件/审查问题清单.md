## 二、扣分原因 + 具体问题 + 对应提升方法

### P0-1（-4分）：发布安全模型仍有硬伤（客户端共享 Key）

- 具体问题：文档仍写“客户端直连 + `.env` 存 Key”。发布后 APK/IPA 可被逆向，存在 Key 泄露与盗刷风险。
- 影响：成本失控、账号风控风险、商业化不可持续。
- 提升方法：
1. 将 PRD 改为“双态策略”：开发态 `.env`，发布态 `BYOK`（用户自填自己的 SiliconFlow Key，本地加密存储）。
2. 明确“发布包禁止内置共享生产 Key”。
3. 若后续公测，再补“极简签名中转（仅签名/限流，不存业务数据）”作为可选项。

### P0-2（-2分）：“100% 零成本”表述过于绝对

- 具体问题：开发与云基础设施可近零，但分发与运维存在刚性成本。
- 影响：目标定义失真，后续预算与预期容易冲突。
- 提升方法：
1. 将“100% 零成本”改为“**云基础设施近零成本**（不含模型调用与分发资质成本）”。
2. 新增“不可避免成本边界”小节：分发资质、签名、真机测试、人力维护。

### P1-1（-1分）：版本策略仍偏静态

- 具体问题：`RN 0.77+ / Expo SDK 52+` 是基线，但缺少“何时升级、如何升级、失败如何回滚”。
- 影响：后期升级时风险集中。
- 提升方法：
1. 新增“版本治理”三段：`Minimum` / `Recommended` / `Rollback`。
2. 把 `skia/reanimated/gesture-handler` 写成兼容矩阵，避免组合冲突。

## 三、用于 Claude Code 的 Vibe Coding 是否还有提升空间？

**有，且主要是“把约束写成机器可验收规则”。**

建议补 5 条：
1. 每个 Phase 增加 DoD：输入、输出、验收命令、失败回滚。
2. 增加自动护栏：lint/静态检查禁止 Base64 写入 store。
3. 增加 `contracts/`：所有 AI 响应先过 schema，再入库。
4. 增加“失败回放”：保存最近 N 次请求摘要（脱敏）。
5. 增加“成本日志”：每次生成记录估算消耗和失败码，用于调优。

## 四、除了硅基流动 API 外，能否做到 0 开销？

**结论：不能绝对 0；可以做到“云基础设施近零”。**

- 可接近 0：数据库、对象存储、业务后端托管（均可端侧化）。
- 不能归零：分发资质、签名链路、兼容测试、维护人力。

建议在 PRD 固化一句：
- “MVP 目标是除模型调用外的云基础设施近零成本，而非总体绝对零成本。”

## 五、Windows 能否完成“完美” vibe coding，且支持各类手机系统？

**结论：可完成高质量跨端开发，但不建议承诺‘完美’和‘全系统通吃’。**

- Windows 本地开发与 Android 调试：可行且成熟。
- iOS：可产出成品，但通常需要远程构建/签名链路，不是纯本地闭环。
- 手机系统支持：建议明确“iOS + Android 主流机型优先”，其他系统按兼容测试声明支持范围。

```md
### 发布安全红线
- 开发态可使用 `.env`；发布态禁止内置共享生产 Key。
- 默认 BYOK：用户输入自己的 API Key，本地加密存储，仅本机使用。

### 版本治理
- Minimum: Expo SDK 52 / RN 0.77
- Recommended: 跟随当前稳定版本（按季度评审）
- Rollback: 升级失败时回退到上一个已验证版本矩阵

### 契约与验收
- AI 接口响应必须先通过 JSON Schema 校验再入库。
- 429/503/超时必须执行指数退避重试，并映射统一 UI 提示。
```
