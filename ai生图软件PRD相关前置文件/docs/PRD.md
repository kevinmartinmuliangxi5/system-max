# AI 生图应用 PRD（MVP）

- 文档状态：可执行评审版
- 更新时间：2026-02-27
- 输入来源：`ai生图软件PRD相关前置文件/AI生图软件（MVP）--idea.md`
- 适用平台：iOS + Android（MVP）
- 成本红线：MVP 运行成本仅包含 SiliconFlow API 调用，不引入其他付费云基础设施

## 1. 产品概述与目标用户

### 1.1 产品概述
本产品是一款移动端 AI 生图/改图应用，核心体验为“多轮对话 + 局部涂抹重绘（Mask Inpainting）”。
MVP 聚焦单机可用闭环：生成 -> 编辑 -> 保存/导出 -> 异常恢复，确保首图转化效率与编辑可控性。

### 1.2 目标用户
1. 社媒创作者：高频配图需求，追求出图速度与迭代效率。
2. 轻设计用户：需要低门槛完成封面、海报、头像等视觉素材。

### 1.3 目标与边界
1. 目标
- 新用户 3 步完成首图生成。
- 支持局部重绘完整链路。
- 关键异常（429/503/超时）可恢复。
2. 非目标
- 不做云数据库、云对象存储、云鉴权。
- 不做 Web 端、复杂运营后台、模型训练/微调。

### 1.4 关键约束
1. 模型能力来源仅使用 SiliconFlow API。
2. 本地持久化使用 `expo-sqlite` + `expo-file-system` + `MMKV`。
3. 发布态禁止内置共享生产 Key，采用 BYOK。
4. AI 响应必须通过 Schema 校验后才能入库与渲染。

## 2. 核心功能列表（P0/P1/P2 优先级）

### 2.1 P0（MVP 上线阻断项）
1. 文生图请求链路
- 输入提示词，调用 `POST /v1/images/generations`，返回图片结果。
2. 图生图请求链路
- 以现有图为输入，执行再生成。
3. 局部重绘链路
- 图片进入编辑器 -> 涂抹 mask -> 输入局部修改指令 -> 返回替换结果。
4. 多轮对话记录
- 保存会话、消息、图片关联关系，支持历史回看。
5. 本地文件落盘
- Base64 结果立即落盘；数据库仅存 `file://` URI 与元数据。
6. 错误重试与统一提示
- 429/503 指数退避最多 3 次；超时重试 2 次；非 429 的 4xx 不重试。
7. 发布态密钥策略
- BYOK 输入、系统安全存储、日志脱敏。

### 2.2 P1（稳定性与可运维增强）
1. 契约校验模块（`contracts/`）
- 维护 `prompt-enhance` / `image-generation` / `error` 响应 Schema。
2. 失败回放日志
- 本地保存最近 N=50 条脱敏失败摘要。
3. 成本日志
- 记录请求类型、状态码、重试次数、耗时、成本估算。
4. OOM 防线
- 禁止 Base64 进入全局 store；编辑前大图降采样。

### 2.3 P2（体验和治理增强）
1. 调试页
- 展示失败摘要并支持脱敏复制。
2. 版本治理自动化
- 维护 Minimum/Recommended/Rollback 矩阵检查。
3. 合规增强
- 本地关键词拦截策略扩展与异常结果处理机制。

## 3. 用户故事 (User Stories)

### 3.1 流程一：首图生成
1. 作为新用户，我希望打开应用后快速输入提示词并生成第一张图，以确认产品价值。
2. 作为创作者，我希望连续发送补充指令优化结果，以减少反复重输提示词。
3. 作为成本敏感的产品方，我希望每次请求记录耗时与估算消耗，以监控运行成本。

### 3.2 流程二：局部重绘
1. 作为用户，我希望在图片上直接涂抹要修改的区域并输入“改成…”，只改局部不破坏整体。
2. 作为用户，我希望编辑过程可平移/缩放并保持流畅，以便精准修改细节。
3. 作为系统维护者，我希望异常返回先经过契约校验再渲染，避免脏数据进入 UI。

### 3.3 流程三：导出与异常恢复
1. 作为用户，我希望结果可保存到相册并带有 AIGC 标识，便于后续发布与合规说明。
2. 作为用户，我希望遇到服务繁忙或超时时系统自动重试并给出下一步提示，不中断创作。
3. 作为排障人员，我希望能查看最近失败请求的脱敏回放信息，便于复盘与重试。

## 4. 非功能需求（性能、安全、可用性）

### 4.1 性能要求
1. 首屏可交互时间（中端机）< 2.5s。
2. 普通生成链路 P50 < 8s。
3. 涂抹编辑目标帧率 55-60 FPS。
4. 图片结果必须“先落盘再渲染”，禁止长生命周期持有 Base64。

### 4.2 安全要求
1. 发布包禁止硬编码共享生产 Key。
2. 发布态仅 BYOK，Key 存储于 iOS Keychain / Android Keystore。
3. 日志禁止输出完整 Key；仅允许掩码显示（前 3 后 2）。
4. 网络与错误日志必须脱敏。
5. 所有 AI 响应必须先 Schema 校验，再入库与渲染。

### 4.3 可用性要求
1. 错误场景必须有统一提示、重试策略与可执行下一步。
2. 关键路径（首图生成、局部重绘、保存导出）在主流机型可稳定完成。
3. 崩溃率目标 < 0.5%。
4. 失败回放与成本日志在离线场景可查看历史记录。

## 5. 验收标准 (Acceptance Criteria)

### 5.1 功能验收
1. 首图路径
- 标准：新用户完成“输入 -> 生成 -> 查看”不超过 3 个关键交互。
- 验证动作：手工走查一次完整路径并记录步骤与结果。
2. 局部重绘路径
- 标准：可完成“选图 -> 涂抹 -> 输入指令 -> 返回结果 -> 替换展示”。
- 验证动作：手工走查并截取前后对比截图。
3. 异常恢复
- 标准：429/503/超时按策略重试并提示；非 429 的 4xx 不重试。
- 验证动作：Mock 或回放异常响应，检查重试次数与提示文案。

### 5.2 工程质量验收
1. 测试
- 命令：`pnpm test`
- 标准：通过，包含重试策略、契约校验、日志脱敏、成本估算相关测试。
2. 类型检查
- 命令：`pnpm typecheck`
- 标准：无类型错误。
3. 静态检查
- 命令：`pnpm lint`
- 标准：无阻断级问题。
4. 构建
- 命令：`pnpm build`
- 标准：构建成功。

### 5.3 数据与安全验收
1. Base64 存储约束
- 标准：数据库与全局状态中不出现大体积 Base64 持久存储。
- 验证动作：静态扫描 store/state 与数据库写入路径。
2. Key 保护
- 标准：发布态无共享生产 Key，日志无明文 Key。
- 验证动作：打包前扫描配置与日志输出点。
3. 契约门禁
- 标准：Schema 校验失败时数据不得进入 UI。
- 验证动作：注入不合法响应，验证拦截与错误流程。

### 5.4 可观测性验收
1. 成本日志
- 标准：每次请求均记录请求类型、状态码、重试次数、耗时、成本估算。
- 验证动作：触发成功与失败请求各一次，检查日志字段完整性。
2. 失败回放
- 标准：仅保留最近 N=50 条，且内容脱敏。
- 验证动作：写入超过上限并验证淘汰策略与脱敏结果。

## 6. 范围外事项 (Out of Scope)

1. 云数据库、对象存储、业务后端托管。
2. Web 端应用形态。
3. 社区广场、团队协作、复杂运营后台。
4. 在线支付与商业化闭环。
5. 模型训练、微调与自建推理服务。
6. 多设备云同步（MVP 阶段不纳入）。
