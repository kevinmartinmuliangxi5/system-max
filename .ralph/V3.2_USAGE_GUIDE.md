# Ralph v3.2 使用指南 - Worker + Reviewer 双进程架构

## 架构概览

```
终端1: Brain (Claude Pro)
  └─ 任务规划、需求分析、架构设计
     ↓
     生成蓝图 (.janus/project_state.json)
     ↓
终端2: Worker (GLM)              终端3: Reviewer (GLM)
  ├─ 读取蓝图                      ├─ 监控代码提交
  ├─ 生成代码                      ├─ 静态分析
  ├─ 提交到queue/pending/          ├─ GLM AI审查
  └─ 等待Review反馈                ├─ 综合决策
     ↓                             ├─ 记录经验
     ←────────────┐                ↓
                  │                给出反馈
                  └────────────────┘
```

## 快速启动

### 1. 使用Brain规划任务（终端1）

```bash
# 运行Brain生成任务蓝图
python brain_v3.py "创建用户注册功能，支持邮箱验证"

# Brain会：
# ✓ 使用Claude Pro分析需求
# ✓ 生成详细规格文档
# ✓ 设计系统架构
# ✓ 创建任务蓝图
```

### 2. 启动Worker + Reviewer（自动化）

```bash
# 一键启动（推荐）
bash .ralph/start_ralph_v3.2.sh

# 选择模式：
#   1) Tmux分屏模式（可视化，推荐）
#   2) 后台模式
```

### 3. 监控运行

**Tmux模式**（推荐）:
```
窗口0: Worker  - 代码生成进度
窗口1: Reviewer - 审查进度（实时可视化）
窗口2: Status  - 系统状态

切换：Ctrl+B, 然后按数字键 (0/1/2)
退出：Ctrl+B, 然后按 D (后台运行)
```

**后台模式**:
```bash
# 查看Worker输出
tail -f .ralph/logs/worker_console.log

# 查看Reviewer输出
tail -f .ralph/logs/reviewer_console.log

# 查看状态
jq -s . .ralph/worker_status.json .ralph/reviewer_status.json
```

### 4. 停止系统

```bash
# 一键停止
bash .ralph/stop_ralph_v3.2.sh

# 或在Tmux中按 Ctrl+C
```

## 工作流程详解

### Brain的工作

1. **需求分析** (Claude Pro)
   - 深入理解用户意图
   - 提出澄清问题
   - 识别功能需求、非功能需求、约束

2. **规格编写** (Claude Pro)
   - 生成详细技术规格
   - 输入/输出定义
   - 业务规则、测试用例

3. **架构设计** (Claude Pro，可选)
   - 选择架构模式
   - 组件设计
   - 技术栈选择

4. **任务分解** (本地)
   - 将任务分解为多个Phase
   - 生成任务蓝图

5. **产出**
   - `.janus/project_state.json` - 任务蓝图
   - `.ralph/specs/*.md` - 规格文档
   - `.ralph/diagrams/*` - 流程图

### Worker的工作

1. **读取蓝图**
   - 监控 `.janus/project_state.json`
   - 解析任务要求

2. **代码生成** (GLM-4.7)
   - 调用Claude Code CLI
   - 使用GLM API生成代码
   - 收集修改的文件

3. **提交审查**
   - 将代码复制到 `.ralph/queue/pending/`
   - 创建上下文信息 `context.json`
   - 等待Reviewer反馈

4. **处理反馈**
   - 如果通过：任务完成
   - 如果不通过：根据反馈修复，重新生成

### Reviewer的工作

1. **监控队列**
   - 实时监控 `.ralph/queue/pending/`
   - 发现新提交立即审查

2. **三层审查**

   **第一层：静态分析** (免费，快速)
   - Python语法检查
   - Pylint代码质量
   - Bandit安全扫描
   - 评分：0-10

   **第二层：GLM AI审查** (包月无限)
   - 逻辑错误检查
   - 性能问题分析
   - 安全漏洞检测
   - 代码规范审查
   - 评分：0-10

   **第三层：综合决策**
   - 综合评分 = 静态40% + GLM60%
   - 通过标准：≥7分
   - 生成详细反馈

3. **经验记录** ✨
   - 记录每次审查结果
   - 提取审查教训
   - 积累到 `.ralph/review_experience.jsonl`
   - 下次审查时查询相似案例

4. **反馈Worker**
   - 通过：移动到 `approved/`
   - 不通过：移动到 `rejected/`，写入反馈
   - Worker读取反馈后修复

## 审查经验系统 ✨

Reviewer会记录每次审查的结果，形成经验库：

```json
// .ralph/review_experience.jsonl
{"timestamp":"2026-02-11T15:30:00","task_id":"task_123","approved":true,"score":8.5,"issues":"无重大问题","lesson":"优秀代码模式：评分8.5，通过审查"}
{"timestamp":"2026-02-11T16:00:00","task_id":"task_124","approved":false,"score":5.2,"issues":"SQL注入风险; 性能问题","lesson":"需要改进：SQL注入风险; 性能问题"}
```

**使用场景**:
- 审查相似代码时自动查询历史
- 显示之前遇到的相似问题
- 提供参考建议

**查看经验**:
```bash
# 查看所有审查记录
cat .ralph/review_experience.jsonl | jq .

# 统计通过率
jq -s 'map(select(.approved == true)) | length' .ralph/review_experience.jsonl

# 查找安全问题案例
grep -i "安全" .ralph/review_experience.jsonl
```

## 配额使用

### Brain (Claude Pro)
- **用途**: 任务规划
- **每任务**: 2-6次调用
  - 简单任务: 2次 (req_dev + spec_writer)
  - 复杂任务: 4-6次 (+ architect + brainstorm)
- **月度预估**: 600-900次 (每天10个任务)
- **配额限制**: 2000次/月
- **使用率**: 30-45%

### Worker (GLM)
- **用途**: 代码生成和执行
- **每任务**: 10-20次调用（包括迭代修复）
- **月度预估**: 3000-6000次
- **配额限制**: 包月无限
- **成本**: 固定

### Reviewer (GLM + 静态)
- **用途**: 代码审查
- **每任务**: 3-5次调用
- **月度预估**: 900-1500次
- **配额限制**: 包月无限
- **成本**: 固定

**总成本**: 双包月订阅，固定成本，零额外费用

## 常见问题

### Q: Worker和Reviewer如何通信？

A: 基于文件队列：
- Worker生成代码 → 写入 `pending/`
- Reviewer监控 `pending/` → 审查
- Reviewer写反馈 → Worker读取

### Q: 审查不通过怎么办？

A: Worker会：
1. 读取详细反馈
2. 根据反馈修复代码
3. 重新生成并提交
4. 循环直到通过

### Q: 如何查看审查历史？

A:
```bash
# 查看经验库
cat .ralph/review_experience.jsonl | jq .

# 查看某个任务的审查详情
ls .ralph/queue/approved/task_*/decision_*.json
```

### Q: 可以只运行Worker或Reviewer吗？

A: 可以，手动启动单个进程：
```bash
# 只运行Worker
bash .ralph/worker_loop.sh

# 只运行Reviewer
bash .ralph/reviewer_loop.sh --live
```
